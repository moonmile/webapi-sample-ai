# 概要設計書 — order-tablet

## 1. システム概要

### 1.1 目的

飲食店（寿司店）向けのセルフオーダー用タブレット Web アプリケーション。  
テーブルに設置されたタブレットから、来店客が自分でメニューを閲覧し、注文・会計・レビュー投稿を完結できるシステム。

### 1.2 システム名

**日経寿司 タブレットオーダーシステム**

### 1.3 対象ユーザー

- **来店客**（一次ユーザー）：メニュー閲覧・注文・会計・レビュー投稿
- **店舗スタッフ**（間接ユーザー）：order-manager でリアルタイム注文確認

## 5. API 連携

### 5.1 エンドポイント一覧

| メソッド | エンドポイント | 呼び出し元 | 認証 | 用途 |
|---------|--------------|-----------|------|------|
| GET | `/api/categories` | カテゴリ画面 | なし | カテゴリ一覧取得 |
| GET | `/api/products?category_id={id}` | カテゴリ画面 | なし | 商品一覧取得 |
| POST | `/api/orders` | カート画面 | X-API-KEY | 注文登録（アイテム単位） |
| GET | `/sanctum/csrf-cookie` | レビュー画面 | なし | CSRF トークン取得 |
| POST | `/api/reviews` | レビュー画面 | CSRF | レビュー投稿 |

### 5.2 認証方式

| API | 認証方式 |
|-----|---------|
| 注文 API | カスタムヘッダー `X-API-KEY`（環境変数 `NEXT_PUBLIC_ORDER_API_KEY`） |
| レビュー API | Laravel Sanctum（CSRF トークン） |
| カテゴリ・商品 API | 認証なし（フォールバックあり） |

### 5.3 エラーハンドリング

- カテゴリ・商品 API エラー時はダミーデータにフォールバック（オフライン対応）
- 注文 API エラー時は画面にエラーメッセージ表示（ユーザー再試行可）
- 認証エラー（401）は専用メッセージ表示

## 6. データ管理

### 6.1 クライアント側ストレージ

| ストレージ | キー | 内容 | ライフサイクル |
|-----------|------|------|--------------|
| sessionStorage | `guests` | 来店人数 | セッション終了まで |
| sessionStorage | `seat_id` | テーブル ID | セッション終了まで |
| localStorage | `cart` | カートアイテム | 注文確定時にクリア |
| localStorage | `orderData` | 直近注文データ | 支払い完了時にクリア |
| Cookie | `XSRF-TOKEN` | CSRF トークン | Laravel Sanctum が管理 |

### 6.2 主要データモデル

#### CartItem

```typescript
interface CartItem {
  id: number;
  name: string;
  price: number;
  quantity: number;
}
```

#### OrderRequestPayload（API 送信）

```typescript
interface OrderRequestPayload {
  seat_id: number;
  product_id: number;
  quantity: number;
}
```

#### OrderData（localStorage 保存）

```typescript
interface OrderData {
  seat_id: number;
  items: Array<{ id: number; name: string; price: number; quantity: number }>;
  total_amount: number;
  guest_count: number;
}
```

#### ReviewRequestPayload（API 送信）

```typescript
interface ReviewRequestPayload {
  rating: number;       // 1-5
  comment: string;
  product_id?: number | null;
  order_id?: number | null;
  seat_id?: number | null;
}
```

## 7. 画面遷移フロー

```
[ホーム]
  │ 来店人数設定 → 「注文開始」
  ▼
[カテゴリ・商品一覧]
  │ カテゴリ選択 → 商品一覧表示
  │ 商品追加 → カートへ
  │ 「カートを見る」
  ▼
[カート]
  │ 数量確認・調整
  │ 「注文を確定する」→ POST /api/orders（アイテム数分）
  ▼
[注文完了]
  ├─ 「注文状況を見る」→ [注文履歴]
  ├─ 「追加注文」→ [カテゴリ・商品一覧]
  └─ 「ホームに戻る」→ [ホーム]

[注文履歴]
  ├─ 「お会計へ」→ [支払い]
  └─ 「追加注文」→ [カテゴリ・商品一覧]

[支払い]
  │ 支払い方法選択 → 「支払いを確定」
  └─ (完了後 localStorage クリア)

[レビュー投稿]（ヘッダーナビゲーションから常時アクセス可）
  └─ POST /api/reviews
```

## 8. 状態管理方針

外部状態管理ライブラリ（Redux・Zustand 等）は使用せず、以下の方針で管理する。

- **ページローカル状態**：`useState` で管理（ローディング・エラー・フォーム値）
- **セッション横断データ**：sessionStorage / localStorage で永続化
- **データの受け渡し**：ページ間はストレージ経由（props や Context を使用しない）

## 11. 制約・前提条件

- テーブル番号は現状ハードコード（`T-001`、`seat_id = 1`）。実運用では URL パラメータ等で動的設定が必要。
- 支払い処理は現状シミュレーション（2秒ウェイト）。実決済連携は未実装。
- 注文履歴はダミーデータと localStorage の組み合わせ。バックエンド API 連携は未実装。
- オフライン時はカテゴリ・商品のみダミーデータで動作可。注文・レビューは API 必須。
